NES Static Recompiler

This project is a static recompiler that translates iNES ROMs into native executables for modern systems. It lifts 6502 machine code into C++, compiles that code against an emulated NES graphics and I/O runtime, and produces working binaries that outperform a pure emulator on the same runtime.

Scope

This project was developed over a 3-month undergraduate dissertation as a proof-of-concept. Audio emulation and games that used memory mapping were deliberately excluded to save time.

Key Technical Highlights

  • Static translation pipeline lifting 6502 machine code into C++
  • Heuristic code discovery for symbol-less binaries
  • Basic-block based translation with overlapping block support
  • Interpreter fallback for unresolved indirect control flow
  • Profile-guided optimisation of interrupt polling and translation seeds

Why Static Recompilation?

Static recompilation is appealing because it allows performance optimisations that would be impossible with emulation. For instance, emulators must implement fetching and decoding behaviour in software that must run for every CPU instruction. This can be completely omitted from recompiled code, which implicitly delegates that functionality to the host hardware.

The tradeoff is that extracting performance from static recompilation requires reducing runtime accuracy and potentially breaking certain games that depend on very specific behaviours of the NES. In practice, however, only a minority of games do this.

Although NES emulators easily run games at full speed on modern devices - even mobile phones - static recompilation could be a solution for improving battery life or bringing games to cheaper and more underpowered devices.

Addressing Challenges

Developing this required overcoming several challenges:

  • Code discovery - identifying executable code in a ROM with no symbol information
  • Pointer translation - remapping ROM and RAM references into host representations
  • Interrupt handling - approximating NES interrupt timing efficiently in host code
  • Self-referential control flow: supporting techniques like instruction overlap and modified return addresses

Binary Disassembly Without Symbols

Disassembling code from a binary with no symbol information is a known hard problem - actually it is equivalent to the halting problem. In the general case, code and data cannot be perfectly distinguished in an unannotated binary. The approach I took therefore had to be heuristic.

Inspired by a paper about ARM→x64 translation, I decided on a liberal method that would translate anything that looked like code as if it were, deliberately preferring overtranslation to undertranslation. Any data mistranslated as code would simply never execute during runtime.

This is the process:

  1. Scan the ROM for any sequence of bytes that could be interpreted as a valid code pointer.
  2. Disassemble a linear sequence of instructions beginning at each of those pointers and ending at a control flow instruction such as a branch or jump.
  3. For each conditional branch encountered, if either or both of its targets haven't been seen before, record them in a new list of pointers. Then repeat step 2.

Code Generation Strategy

I translated code as basic blocks. A block starts at a control-flow target and ends at the next control-flow instruction (branch, jump, call, return, interrupt return, etc). Each block is emitted as a C++ function, allowing control flow to become direct C++ calls between blocks instead of a fetch/decode loop.

To support ROM tricks like overlapping instruction streams, I allowed overlapping blocks. This means some bytes may be translated more than once, but it preserves correctness when the same address is reached in different execution contexts.

At runtime, every translated block follows the same structure: it updates the emulated environment with accumulated CPU cycles, checks for an interrupt, executes the translated instructions, increments the cycle counter by the block’s total cost, and tail-calls the next block. CPU state, RAM, and ROM live in shared runtime variables, while instruction behavior is implemented through small helper functions for common CPU operations.

For indirect control flow, the runtime uses a 16-bit lookup table mapping NES addresses to translated block entry points. If a target has no translated block, execution falls back to an instruction emulator until control reaches translated code again. This keeps unusual patterns, including modified return addresses, working without needing fully dynamic recompilation.

Interrupt Simulation Tradeoff

On real NES hardware, interrupts are polled-for after each instruction. Reproducing that literally in translated code would require an interrupt test after every emitted instruction, which adds branch overhead almost everywhere and erodes the speed benefits of static recompilation.

I chose to poll for interrupts at block boundaries instead. In practice that means checking after each translated jump/branch. Every block begins with an interrupt check, and in the dispatcher the same check exists before fallback emulation continues.

This is an accuracy tradeoff: interrupt latency can be up to one block. But in testing it did not cause visible issues in the games I tried, and it kept the generated code much simpler and faster.

Profile-Guided Optimisations

Because translation quality depends on what code actually executes, I added lightweight profiling hooks so runtime behavior can feed back into later translation passes.

Two profiles were especially useful:

  1. Jump-target profile: record which ROM addresses were actually reached, then use those as direct translation seeds instead of relying only on static pointer heuristics.
  2. Interrupt-location profile: record where interrupts were observed, then keep expensive interrupt checks/environment sync only where interrupts actually occur.

These profiles are emitted by optionally enabled runtime instrumentation and can be written to files for reuse by the translator. In many games, interrupts occur in a small number of hot wait loops, so this can remove a large amount of per-block overhead in the rest of the code.

Results

To evaluate the translator, I benchmarked the same mapper 0 homebrew shown in the video, *GunTneR*, on a local Ubuntu 24.04.4 system with a Ryzen 7 9800X3D CPU. I ran 600 frames with no input, allowing the game to settle into its attract mode, and recorded the time from one NMI to the next while excluding rendering overhead and any sleep used to maintain 60 Hz. I compared three configurations on the same runtime: pure emulation, static recompilation, and static recompilation with the profile-guided optimisations described above.

Even though the runtime still emulates the NES PPU, I/O hardware, and falls back to an interpreter for unresolved control flow, the statically recompiled build still achieved a 6.48% speedup, increasing to 8.49% with profile-guided optimisation. The results show that even a relatively simple static recompiler can produce a measurable performance improvement over a pure emulator running on the same runtime.

Reflection

This project gave me a much clearer view of what static recompilation can and cannot do. Code discovery in an unannotated binary is inherently heuristic, indirect control flow cannot be resolved perfectly in the general case, and translation becomes harder as the gap widens between the source and target execution environments. In practice, that meant accepting that recompilation could not replace emulation entirely: even in the translated build, I still needed to emulate the PPU and I/O, and to retain a fallback emulator for cases that could not be handled statically.

Those constraints shaped several deliberate compromises. I limited the scope to games that didn't use memory mapping and chose block-level interrupt polling rather than cycle-accurate execution, which kept the translator tractable and the generated code simple enough to optimise. Even with those simplifications, the design succeeded in its main aim: it ran a range of licensed and homebrew mapper 0 games with few visible accuracy issues, and it delivered a measurable speedup over emulation on the same runtime.

If I were extending the project, the most useful next steps would be memory mapper support, a faster or GPU-accelerated PPU implementation, and more aggressive optimisation of how CPU state is represented in generated code. More broadly, I am interested in continuing to improve my knowledge and skills around the intersection of recompilation and reverse engineering: building tools that recover, analyse, and transform existing binaries while accounting for the messy constraints of real systems.
